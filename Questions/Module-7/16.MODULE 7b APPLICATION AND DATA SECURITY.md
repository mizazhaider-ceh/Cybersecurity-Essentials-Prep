***

**Question 31** (üî¥ **CRITICAL - DEFINITION!**)

**DevSecOps:**

**Definition:**

- Methodology (based on DevOps) used in software development with **integration of cybersecurity**

**Key differences from SDLC:**

- Integrates phases in **continuous cycle** of:
    - Development
    - Integration
    - Testing
    - Deployment
    - Monitoring
- Emphasizes **close collaboration** between:
    - Development teams
    - Security teams
    - Operations teams

**What does DevSecOps integrate?**

**Answer:** Development + Security + Operations

**DevSecOps Key Features:**

**Automation:**

- Use of automated security testing and controls within **CI/CD pipeline**
- CI/CD = Continuous Integration/Continuous Delivery

**Continuous Feedback:**

- Security feedback is continuously collected and processed

**Continuous Cycle:**

- Not linear like SDLC - continuous loop!

**True or False:** "DevSecOps uses continuous integration and delivery (CI/CD)."

**Answer:** True

***

**Question 33** (üî¥ **CRITICAL - DEV PHASE ACTIVITIES!**)

**DevSecOps - Dev Phase Activities:**

1. **Threat Modelling:**
    - Identify and analyze potential threats early in process
    - **Tool example:** Threat Dragon (OWASP)
2. **Secure Coding:**
    - Follow guidelines for secure coding
    - Conduct code reviews
3. **Security as Code (SaC):**
    - Security measures considered part of code
    - Subject to automation and version control
4. **SAST (Static Application Security Testing):**
    - **"White-box"** testing method
    - Analyzes **source code** to identify vulnerabilities
    - **WITHOUT running application**
    - **Tool example:** SonarQube

**Match:** "Analyzing source code without running the application" = ?

**Answer:** SAST (Static Application Security Testing)

***

**Question 34** (üî¥ **CRITICAL - MORE DEV ACTIVITIES!**)

**DevSecOps - More Dev Phase Activities:**

5. **DAST (Dynamic Application Security Testing):**
    - **"Black-box"** testing method
    - Examines **running application** to detect vulnerabilities
    - As an attacker would
    - **Tool examples:** OWASP ZAP, Burp Suite, Integriti
6. **IAST (Interactive Application Security Testing):**
    - **"Grey-box"** testing method
    - Detects vulnerabilities by monitoring application **in real-time**
    - Uses **embedded agents**
7. **Pentesting:**
    - Uses DAST tools + **manual verification**
8. **Digital Sign:**
    - Signing code as **proof of origin** for user

**Match:** "Testing running application as attacker would" = ?

**Answer:** DAST (Dynamic Application Security Testing)

## **PART E: DATABASE SECURITY (Questions 116-145)**


***

**Question 116** (üî¥ **CRITICAL - TYPICAL ISSUES!**)

**Data in Databases - Typical Issues:**

1. **SQL Injection:**
    - Attackers inject malicious SQL code to access/manipulate data
2. **Unpatched Software:**
    - Outdated database software contains exploitable vulnerabilities
3. **Insufficient Access Control:**
    - Lack of strict access controls ‚Üí unauthorized access to sensitive data
4. **Weak Authentication:**
    - Weak passwords or lack of MFA ‚Üí increased breach risk
5. **Unencrypted Data:**
    - Data not encrypted ‚Üí vulnerable to interception/theft
6. **Misconfiguration:**
    - Errors in database configuration ‚Üí security breaches/unauthorized access

**Most common attack?**

**Answer:** \#1 - SQL Injection

***

**Question 117** (üî¥ **CRITICAL - CONTROLS!**)

**Data in Databases - Controls:**

1. **Encryption:**
    - Encrypt data both **at rest** and **during transmission**
2. **Access Control:**
    - Use **RBAC** (Role-Based Access Control)
    - Determine who has access to what data
3. **Authentication:**
    - Implement strong authentication methods (MFA)
4. **Audit and Logging:**
    - Keep detailed logs of database activity
    - Detect suspicious activity
5. **Patch Management:**
    - Database software up-to-date with latest security patches
6. **Backups:**
    - Regular backups
    - Test recovery procedures

**What does RBAC stand for?**

**Answer:** Role-Based Access Control

***

**Question 118** (üî¥ **CRITICAL - BEST PRACTICES!**)

**Data in Databases - Best Practices:**

1. **"Least Privilege":**
    - Give users only **minimum access rights** they need
2. **Segregation of Duties:**
    - Segregate roles/responsibilities to reduce internal threat risk
    - Example: **Database Administrator vs Database Auditor**
    - Only auditor can delete/modify logs
3. **Database "Firewalls":**
    - Block malicious SQL injections and other attacks
4. **Regular Audits:**
    - Conduct regular audits and pentests
    - Identify and fix vulnerabilities
5. **Intrusion Detection Systems (IDS):**
    - Detect unauthorized access attempts
6. **Data Masking:**
    - Hide sensitive data in non-production environments

**Why separate Database Administrator from Database Auditor?**

**Answer:** Segregation of duties - prevent admin from tampering with logs

***

**Question 119** (üî¥ **CRITICAL - DATABASE-LEVEL CONTROLS!**)

**Data in Databases - Controls at Database Level:**

1. **Referential Integrity:**
    - Ensures references between tables are valid
    - Prevents inconsistent data
2. **Entity Integrity:**
    - Each record has unique key
    - Prevents duplication
    - Ensures accurate identification
3. **Validation of Input:**
    - Checks entered data for accuracy/completeness
    - **Prevents SQL injections** and other attacks
4. **Defined Data Fields (Schema):**
    - Determines structure and type of data
    - Prevents entry of incorrect/harmful data

**Which control specifically prevents SQL injection at database level?**

**Answer:** \#3 - Validation of Input

***

**Question 161** (üî¥ **CRITICAL - GDPR BASICS!**)

**GDPR (General Data Protection Regulation):**

**What is it:**

- **EU General Data Protection Regulation**
- Effective since **May 25, 2018**
- Deals with protection of **PERSONAL DATA**

**Legal status:**

- A **regulation** = immediately binding

**In Belgium:**

- **AVG** = Algemene Verordening Gegevensbescherming

**Personal data definition:**

- Anything that can **identify a natural person**

**Requirement:**

- All organizations must **report certain data breaches** to relevant supervisory authority
- In Belgium: **"Gegevensbeschermingsautoriteit"**

**When was GDPR effective?**

**Answer:** May 25, 2018

***

**Question 162** (üü° Medium - **GDPR ROLES!**)

**GDPR Roles:**

1. **Controller:**
    - Entity that determines **purpose and means** of processing personal data
2. **Processor:**
    - Entity that processes personal data **on behalf of controller**
3. **Data Protection Officer (DPO):**
    - Person designated to **oversee GDPR compliance** within organization
4. **Data Subject:**
    - The **person whose personal data** is being processed

**Match:** "Person whose email address is stored in company database" = ?

**Answer:** Data Subject

***

**Question 163** (üü° Medium - **MATERIAL SCOPE!**)

**GDPR Material Scope - Does NOT apply to:**

a) Activity falling **outside scope of Union law**

b) Member States carrying out activities within scope of **Chapter 2 of Title V of TEU**

c) Natural person in course of **purely personal or household activity**

d) Competent authorities for purposes of:

- Prevention, investigation, detection, prosecution of **criminal offences**
- Execution of criminal penalties
- Safeguarding against/prevention of threats to public security

**Example of personal/household activity exempt from GDPR:**

**Answer:** Personal contact list on your phone for family/friends

***

**Question 164** (üî¥ **CRITICAL - DEFINITIONS!**)

**GDPR Definitions:**

**'Personal Data':**

- Any information relating to **identified or identifiable natural person** ('data subject')
- Identifiable person can be identified directly/indirectly by:
    - Name, ID number, location data, online identifier
    - One or more factors specific to physical, physiological, genetic, mental, economic, cultural, or social identity

**'Processing':**

- Any operation performed on personal data
- Whether or not by automated means
- Such as: collection, recording, organisation, structuring, storage, adaptation, alteration, retrieval, consultation, use, disclosure, dissemination, alignment, combination, restriction, erasure, destruction

**Is storing email addresses considered processing?**

**Answer:** Yes (storage is explicitly listed as processing)

***

**Question 165** (üî¥ **CRITICAL - GDPR PRINCIPLES PART 1!**)

**GDPR Principles:**

1. **Lawfulness, Fairness, Transparency:**
    - Data must be processed in lawful, fair, transparent manner
2. **Purpose Limitation:**
    - Collected for specified, explicit, legitimate purposes
    - Not further processed incompatibly with those purposes
3. **Data Minimization:**
    - Adequate, relevant, limited to what's necessary
4. **Accuracy:**
    - Data must be accurate and up-to-date
    - Incorrect data must be corrected or deleted

**Match:** "Only collecting email addresses when email contact is needed" = ?

**Answer:** Data Minimization principle

***

**Question 166** (üî¥ **CRITICAL - GDPR PRINCIPLES PART 2!**)

**GDPR Principles (continued):**

5. **Storage Limitation:**
    - Data should not be kept **longer than necessary** for purposes
6. **Integrity and Confidentiality:**
    - Data must be processed ensuring security
    - Including protection against:
        - Unauthorized/unlawful processing
        - Accidental loss, destruction, damage
7. **Accountability:**
    - Organizations must **demonstrate compliance** with GDPR principles

**Match:** "Deleting customer data after contract ends" = ?

**Answer:** Storage Limitation principle

***

**Question 167** (üî¥ **CRITICAL - PERSONAL DATA BREACH!**)

**GDPR - Personal Data Breach:**

**Definition:**

- **Breach of security** leading to:
    - Accidental or unlawful destruction
    - Loss
    - Alteration
    - Unauthorized disclosure of, or access to
    - Personal data transmitted, stored, or otherwise processed

**Legal Obligation:**

- Report **not later than 72 hours** after becoming aware of breach

**Report to:**

- Competent authorities (**Gegevensbeschermingsautoriteit** in Belgium)

**Exception:**

- Unless **unlikely to result in risk** to rights/freedoms of natural persons

**Within how many hours must breach be reported?**

**Answer:** 72 hours
